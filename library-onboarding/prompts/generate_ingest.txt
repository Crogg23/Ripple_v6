Write a Python ingestion function for the source below. It feeds a RAW landing
zone: an exact, untyped mirror of the source (every value as a string).

SOURCE
  Name:           {name}
  URL:            {url}
  Access pattern: {access_pattern}
  Auth:           {auth_type} ({auth_notes})
  Auth env var:   {auth_env_var}
  Format:         {data_format}
  Rate limits:    {rate_limits}
  Schema:
{schema}

FOREMAN FEEDBACK (apply this if not "(none)"):
  {feedback}

REQUIREMENTS
  - Use the EXACT source URL shown above (also in context["url"]). Do NOT swap in
    a different host, endpoint, or API version from memory -- look-alike hosts are
    a common trap. The URL above is authoritative; your prior is not. Keep any
    query parameters it already has.
  - Output ONLY one Python code block. No explanation before or after.
  - Define exactly: def fetch_data(context): ... returning a pandas.DataFrame.
  - Store EVERY column as a string -- no type casting. Raw is a verbatim mirror;
    dbt staging does the casting later. (df = df.astype(str) is fine.)
  - Do NOT add _INGESTED_AT / _SOURCE_RUN_ID / _SRC_SHA256 -- the caller does that.
  - Do NOT connect to Snowflake. Just return the DataFrame.
  - Stash the raw source bytes and filename for content hashing + provenance:
        context["source_bytes"] = resp.content      # the exact bytes you downloaded
        context["source_file"]  = "all_month.csv"    # filename, if a file
    For paginated APIs, accumulate the raw response bodies (e.g. join them) into
    context["source_bytes"] so the SHA-256 reflects the real payload.
  - `context` also has: "url", "source_name", "auth_type", and "env" (a dict --
    read API keys from here, e.g. context["env"].get("FRED_API_KEY")).
  - When auth is required, read the key from EXACTLY the env var named on the
    "Auth env var" line in SOURCE: context["env"].get("<that name>"). Do not
    invent a different variable name. If that line is blank, "(none)", or an
    un-filled template token, fall back to the conventional name implied by the
    auth notes.
  - Use `requests` for HTTP. Loop for pagination. Respect rate limits with
    time.sleep() derived from the documented limit.
  - User-Agent: ALWAYS set a descriptive User-Agent header on every request. Some
    government sources REQUIRE it and return 403 without a real contact UA -- SEC
    EDGAR (www.sec.gov / data.sec.gov) is the classic case: it rejects a missing or
    generic agent. Use e.g. {"User-Agent": "Ripple Onboarding Agent (contact:
    ripple-data@example.com)"} for APIs and bulk files. (For bot-protected SCRAPE
    pages a browser-like "Mozilla/5.0 ..." UA is the right choice instead.)
  - Bounded snapshot: if the source is huge or effectively unbounded (a search API
    over millions of rows, an Elasticsearch from/size endpoint, etc.), do NOT page
    through the whole dataset. Respect the limit/size/page already in the URL and
    fetch a single bounded page (or a small capped number of pages). Many APIs also
    reject deep pagination (offset > 10,000).
  - Incremental sources (the config sets load_mode=incremental): context["since"]
    holds the high-water mark -- the latest cursor value already loaded -- or None on
    the first run. If it is set, fetch ONLY records at/after it (e.g.
    date_received_min = context["since"][:10]); if None, fetch a bounded initial
    backfill. Return just that slice -- the caller APPENDS it (landing is an append
    log; staging dedups on the primary key). Do not try to mirror the whole source.
  - Chunked sources (the config sets load_mode=chunked): the file is too big to hold
    in memory (multi-GB CSV/ZIP). Make `fetch_data` a GENERATOR that `yield`s pandas
    DataFrames -- one batch of ~context["chunk_rows"] rows at a time -- NEVER building
    the whole thing. Stream it:
        * Direct CSV: `for chunk in pd.read_csv(url, dtype=str, chunksize=context["chunk_rows"]): yield chunk`
        * Big ZIP: stream-download to a temp file (requests stream=True, iter_content to
          disk -- do NOT hold the bytes in memory), then `zipfile.ZipFile(path).open(member)`
          and `pd.read_csv(member, dtype=str, chunksize=context["chunk_rows"])`, yielding chunks.
      Honour the cap; the caller handles resume:
        * Always yield the file FROM THE START -- do NOT try to skip rows yourself.
          The caller drops rows already landed (it passes context["resume_from_row"]
          for reference only; you can ignore it).
        * context["chunk_max_rows"]: if > 0, stop yielding once you've produced that many
          rows total (a bounded cap); if 0, stream the whole file.
      Do NOT set context["source_bytes"] (the caller hashes each chunk). Read everything
      as strings (dtype=str). The caller stamps provenance + writes each chunk to Snowflake.
  - Scrape sources (access_pattern = scrape): there is no clean file/API. Use
    requests + BeautifulSoup (both available). Find the data-bearing element -- an
    HTML <table> (pandas.read_html(resp.text) is often enough), or a repeated listing
    of rows/cards -- and extract ONE ROW PER RECORD into the DataFrame. Stash the page
    bytes: context["source_bytes"] = resp.content. Send a browser-like User-Agent
    header (e.g. "Mozilla/5.0 ..."); some sites 403 non-browser agents.
  - PAGINATE listing/search/portal scrapes -- this is the #1 scrape bug. A portal that
    shows results almost ALWAYS spans many pages; grabbing only page 1 lands a handful
    of rows and misses ~all the data. LOOP over pages until a page yields no new records
    (or you hit a sane cap of ~50-100 pages): follow the "next" link, or increment the
    page/offset/start parameter (?page=N, &offset=N, &start=N, &from=N), or post the
    pager form. Treat a result of only a few rows as a SIGNAL you stopped at page 1 --
    re-check for a next-page mechanism before returning. Accumulate every page's records
    into one DataFrame and join the page bytes into context["source_bytes"].
  - Bounded crawl: if the records are spread across linked sub-pages (an index page
    linking to per-entity pages/files), fetch the index, extract the matching links,
    then follow at most ~25 of them (a bounded sample) and parse each into rows.
    NEVER crawl unboundedly. Accumulate the fetched HTML into context["source_bytes"].
  - JS-rendered / bot-protected sources (access_pattern = scrape_js): plain requests
    gets back an empty SPA shell or a bot-challenge page ("Just a moment...",
    Cloudflare), NOT the data. Use the headless browser the caller injected:
        html = context["render"](url)                       # runs JS, clears the challenge
        # or wait for the data element to appear before reading (most robust):
        html = context["render"](url, wait_selector="table.results")
    `render(url, wait_selector=None, scroll=False, timeout_ms=45000)` returns the
    fully-rendered HTML as a string. Then parse it EXACTLY like a static page
    (BeautifulSoup(html, "html.parser") or pandas.read_html(html)) -- one row per
    record. Set context["source_bytes"] = html.encode("utf-8"). Pass scroll=True for
    infinite-scroll / lazy-loaded lists. For a bounded crawl of JS sub-pages, call
    render() on the index, extract the links, then render() at most ~25 sub-pages.
    Do NOT import playwright yourself -- only ever use context["render"].
  - Never return raw HTML as data: if you cannot find structured records, raise a
    RuntimeError describing the page (the loader rejects single-column HTML dumps).
  - For bulk CSV/ZIP: download, parse with pandas (io.BytesIO / zipfile as needed).
  - For paginated APIs that return nested JSON, store each record as one row
    (json.dumps the record into a single column if the shape is irregular).
  - If a required API key is missing from context["env"], raise RuntimeError
    naming the exact env var to set. Fail loudly on auth errors or empty results.
  - Keep imports at the top of the code block.
